# Lhama Chat — Revamp Plan

*Prepared July 3, 2026*

## Where things stand today

The app is a SvelteKit 5 SPA (`adapter-static`) shipped two ways: as a web build served behind nginx, and as a Tauri 2 desktop app for macOS, Windows, and Linux (with iOS/Android icon assets already scaffolded, though no mobile build target is wired up yet). There's no component library or CSS framework — every component hand-rolls its own `<style>` block against a shared set of CSS custom properties in `static/global.css` (light/dark tokens, a `.btn`/`.card`/`.alert` mini-kit). That token layer is a reasonable foundation, but it's inconsistently applied: some components use the shared variables, others hardcode colors (`MessageGif.svelte` still has `#f5f5f5`, `#007bff`, etc. instead of `var(--accent)`), and there's no spacing, typography, or elevation scale beyond what each component author improvised. Real-time delivery is duplicated across two independent transports — STOMP-over-SockJS (`websocket.ts`) and MQTT (`mqtt.ts`) — selected at build time via `PUBLIC_REALTIME_MODE`, which doubles the surface area for reconnect/backoff bugs for no clear runtime benefit. Test coverage exists (Vitest + Testing Library) but is thin relative to the app's size — mostly auth/store logic, nothing for the chat page itself, the realtime layer, or the Tauri-specific code paths.

Two concrete bugs were fixed as part of this pass (details below): sessions being wiped on ordinary network hiccups at Tauri cold start, and message links silently failing to open in the desktop app. Both were symptoms of a broader pattern worth calling out for the roadmap: **Tauri-specific behavior is patched in ad hoc per call site** (`'__TAURI_INTERNALS__' in window` checks are scattered across five-plus files) rather than centralized behind a small platform-abstraction layer. That's the root cause of both bugs — someone added the correct Tauri-aware link-opening logic to one call site (`+page.svelte`) and later two more call sites were added (`LinkPreview.svelte`, `MessageGif.svelte`) without it. It'll keep happening until there's one place that owns "how do I open a URL / send a notification / navigate" per platform.

## Bugs fixed this session

**Forced re-login after closing the Tauri app.** `checkAndRefreshToken()` runs on every cold start whenever the stored token is within 48 hours of expiring, which for most token lifetimes is *most* app launches. Its failure path called `logout()` (wiping the stored session) on *any* refresh error — not just a rejected token, but also a plain network error, which is exactly what happens when a Tauri app's webview starts making requests before the OS network stack is fully up, or if the auth server's CORS policy doesn't recognize the Tauri origin the way it recognizes `chat.lhamacorp.com` (there's already a hint of this in `authRedirect.ts`, which special-cases the login-callback origin for Tauri). The git history shows a previous attempt at this fix (`ea57a15`, reverted 20 minutes later) that swapped `localStorage` for `@tauri-apps/plugin-store` — but that change never touched the actual bug, since the same "logout on any error" catch block would have remained. The real fix: `src/lib/api/auth.ts` now throws a typed `TokenRefreshError` that carries the HTTP status when the server actually responded, and `src/lib/stores/auth.ts` only calls `logout()` when that status is 401/403 (a genuinely rejected token). Any other failure — offline, DNS not ready, a 5xx, a CORS rejection — leaves the existing, not-yet-expired session intact and simply retries later. Covered by new regression tests in `src/tests/app.test.ts`.

**Message links not opening the browser in the Tauri app.** `window.open()` doesn't reliably do anything useful inside a Tauri webview — there's no browser chrome to pop a tab into. One call site (`confirmAndOpenLink` in the chat page) already knew this and correctly branched to `@tauri-apps/plugin-opener`'s `openUrl()` when running under Tauri. Two others didn't: `LinkPreview.svelte`'s rich preview cards (YouTube, GitHub, Twitter/X, Spotify, Instagram, Amazon, and Lhamacorp links — i.e. every platform the app auto-detects and renders as a card) called `window.open()` directly, and `MessageGif.svelte`'s fallback "couldn't load, here's the raw link" anchor used a plain `target="_blank"`. Both now route through the same Tauri-aware opener used elsewhere. Given how central link previews are to the message feed, this was likely the single most common way a Tauri user would hit "clicking a link does nothing."

## Design system

Right now "design" lives as tribal knowledge spread across ~15 components' `<style>` blocks. The highest-leverage move is formalizing what's already implicit: promote the token set in `global.css` into a complete scale (spacing, radii, font sizes, and elevation are already partially there; add a motion/duration scale and a semantic color scale for states like unread/mentioned/muted), then do a pass replacing hardcoded values (the colors in `MessageGif.svelte`, the ad hoc pixel values scattered through modal and dropdown styles) with references to it. On top of that, pull the repeated patterns — modal-overlay/modal-content, dropdown menus, the loading spinner, empty states, toast/alert banners — into shared Svelte components instead of re-implementing the same markup and CSS in `+page.svelte`, `chat/[chatId]/+page.svelte`, and others each time a modal is needed. This isn't a rewrite: the visual language (the glassmorphic dark theme, the Inter typeface, the indigo accent) is good and worth keeping — it just needs to stop being copy-pasted.

## Mobile support

The app already has real-viewport handling (`visualViewport` resize listeners for the on-screen keyboard, `env(safe-area-inset-bottom)`, `100dvh`, touch-action tuning) and responsive breakpoints at 768px/480px in most components, which is more than a lot of "we'll add mobile later" codebases have. What's missing is treating mobile as a first-class target rather than a squeeze-to-fit of the desktop layout: touch targets in the chat page (reaction picker, reply swipe, message long-press menu) need audit against the 44px minimum-tap-target guideline; the emoji autocomplete and picker popovers are positioned assuming mouse hover and will need a bottom-sheet pattern on small screens; and there's no pull-to-refresh or swipe-back gesture despite the app already fighting `overscroll-behavior` to prevent the browser's native versions of those. Longer term, since the Tauri config already has iOS/Android icons generated, actually wiring up `tauri android init` / `tauri ios init` as real build targets (even just for internal testing) would validate the mobile CSS work against a real mobile webview instead of just a resized desktop browser — Chrome DevTools device emulation and WKWebView/mobile Chrome have enough behavioral differences (viewport units, momentum scrolling, keyboard-avoidance) that they're not a substitute for testing on the target. A PWA manifest for the web build (there currently isn't one — `app.html` links a favicon but no `manifest.json`) would also let mobile web users install a home-screen icon without needing the native build at all.

## Web/Tauri feature parity

This is the category the two fixed bugs came from, and the fix for both should generalize: introduce a single `src/lib/platform/` module that wraps every capability that behaves differently across web and Tauri — opening external links, showing notifications, checking "am I in Tauri," reading/writing persisted auth state — behind one function per capability. Every one of the five-plus scattered `'__TAURI_INTERNALS__' in window` checks in the codebase today (`+page.svelte`, `LinkPreview.svelte`, `MessageGif.svelte`, `authRedirect.ts`, `osNotification.ts`, `chatNotifications.ts`, `updater.ts`) becomes a call into that module instead, so the next time a new "open a link" or "show a notification" call site is added, it inherits the correct behavior automatically rather than needing someone to remember the Tauri special case. Beyond the two bugs already fixed, worth auditing with the same lens: deep-linking (does clicking a `chat.lhamacorp.com/chat/xyz` link on the user's OS actually route into the running desktop app, or open a second web tab?), single-instance behavior (does launching the app while it's already running focus the existing window, or spawn a duplicate?), and the system tray / minimize-to-tray behavior that desktop chat apps are typically expected to have and this one currently doesn't (closing the window fully quits rather than backgrounding). The updater (`tauri-plugin-updater`) and OS notifications are already implemented Tauri-aware and look solid; they're good reference implementations for how the rest of the platform module should be structured.

## Architecture and reliability

Two things would pay for themselves quickly. First, pick one realtime transport (MQTT looks like the newer/preferred one given `PUBLIC_REALTIME_MODE` defaults and recent commits) and retire the other, or at minimum extract both behind a single interface so the chat page doesn't have five `if (realtimeMode === 'mqtt')` branches — that duplication is exactly the kind of thing that produces "works over websocket, broken over mqtt" bugs. Second, the test suite needs to grow into the areas it currently skips entirely: the chat page component itself (2000+ lines, zero direct tests), the realtime connection/reconnect logic, and — now that this pattern has bitten twice — the platform-abstraction module once it exists, so a future "add a new Tauri-aware call site" change is caught by a test rather than discovered by a user. The `.github/workflows/build.yml` CI already runs tests and a web build on every PR; extending it to also run `tauri build` (or at least `cargo check`) on PRs that touch `src-tauri/` would catch Rust-side regressions before they reach a release.

## Suggested sequencing

1. **Now (done this session):** the two reported bugs — auth persistence and link opening — are fixed and covered by tests.
2. **Next:** extract the platform-abstraction module and migrate the existing scattered Tauri checks onto it; add the PWA manifest; do the design-token cleanup pass (replace hardcoded colors, extract shared modal/dropdown components).
3. **After that:** mobile touch-target and gesture audit, real iOS/Android build targets for testing, tray/minimize and single-instance/deep-link behavior on desktop, realtime-transport consolidation.
4. **Ongoing:** grow test coverage alongside each of the above rather than as a separate cleanup pass — particularly the chat page and the new platform module.

## One thing worth deciding

The realtime-transport question (websocket vs. MQTT) affects how much of the architecture work is worth doing now versus after that's resolved, so it's worth a deliberate call rather than defaulting. I'm happy to dig into the tradeoffs (broker infra you're already running, message-ordering/delivery guarantees, reconnect behavior on mobile networks) whenever you want to make that decision.
